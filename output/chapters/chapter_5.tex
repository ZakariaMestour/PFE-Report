\section{Introduction}
\label{sec:ch5_introduction}

The previous chapters described the platform from the outside inward: business context, technical architecture, backend domain model, and frontend user flows. This chapter focuses on the academic project assigned during the internship: the Disbursement Module. In the SCF chain, this module is the moment where a validated financing intention becomes an operational request. The user no longer configures a product or uploads invoice data only for preparation. The user selects eligible invoices, checks the financing terms, reviews the generated accounting and repayment information, and submits the request into a governed lifecycle.

The module was not developed in isolation from the real platform. It was studied alongside daily sprint work on the Supply Chain Finance platform at Adria. At the same time, the project context imposed a clear boundary: the main platform had not yet reached the sprint milestone where the Disbursement Module would be injected natively into the architecture. The work therefore combined functional analysis, target design, UI-oriented implementation, service-contract preparation, and weekly validation with the Product Owner.

\section{Context and Motivation}
\label{sec:ch5_context_motivation}

\subsection{Position in the SCF Value Chain}
\label{subsec:ch5_value_chain}

Disbursement appears after several upstream conditions have been prepared. A product must exist. A program must bind the anchor, counterparties, finance limits, currencies, dates, fees, disbursement rules, and repayment rules. Invoices or cashflows must be uploaded, validated, and associated with the program. Only then can the financing request be assembled.

Figure~\ref{fig:ch5_value_chain_map} shows the target position of the module inside the SCF value chain. The Disbursement Module consumes program rules, eligible invoices, product context, and fee configuration. Its outputs are not limited to a simple payment request. They include accounting entries, a repayment schedule, and an approval lifecycle state.

    \centering
    \includegraphics[width=\textwidth, keepaspectratio]{images/ch5_value_chain_map.png}
    \caption{Disbursement Module position inside the SCF value chain}

This position explains why the module is sensitive. A wrong program selection, an ineligible invoice, a mistaken finance percentage, or an incorrect fee basis can affect the bank's exposure and the customer's payment schedule. The interface and backend contract must therefore guide the user step by step instead of asking for an unstructured manual request.

\subsection{Why the Module Was Treated as Parallel Work}
\label{subsec:ch5_parallel_work}

During the internship, the SCF platform was still moving from early sprints toward a broader functional perimeter. The core team had not yet reached the integration sprint planned for the Disbursement Module. For this reason, the academic project was organized as parallel work. It advanced during available sprint gaps, after assigned Jira tickets, and through weekly refinement sessions with the Product Owner.

This setup shaped the work. The module had to remain realistic enough to match the existing platform, but flexible enough to absorb future backend decisions. The existing codebase already contained useful anchors: \texttt{FinanceDisbursementModal}, its six panes, \texttt{invoiceDisbursementService}, \texttt{financeDisbursementService}, and the TypeScript model \texttt{FinanceDisbursement}. These elements provided a visible target for the user flow, even while the native backend integration still needed a finalized contract.

\label{subsec:ch5_existing_platform_relation}

\section{Design}
\label{sec:ch5_design}

\subsection{Module Architecture}
\label{subsec:ch5_module_architecture}

The target architecture follows the platform's established pattern. The React interface uses a multi-pane wizard. The wizard calls typed service functions. The service functions use the shared Axios client, which sends authenticated requests through the gateway. The gateway routes \texttt{/scf/} calls to the SCF service. The backend then validates and persists the disbursement request using the program, invoice, fee, accounting, and repayment context.

The existing frontend already reflects this structure through \texttt{FinanceDisbursementModal}. Its panes form a natural module boundary:

\begin{itemize}
    \item \texttt{ProgramProductSelectionPane}: selects the program and product context;
    \item \texttt{InvoiceSelectionPane}: loads and selects eligible invoices;
    \item \texttt{FinanceDetailsPane}: captures financing terms and fee breakdown;
    \item \texttt{AccountingEntriesPane}: displays debit and credit lines;
    \item \texttt{RepaymentDetailsPane}: prepares the repayment schedule;
    \item \texttt{ReviewSubmitPane}: reviews and submits the request.
\end{itemize}

\section{Implementation}
\label{sec:ch5_implementation}

\subsection{Frontend Implementation}
\label{subsec:ch5_frontend_implementation}

The frontend side is the most visible part of the module in the available project context. The wizard structure is already aligned with the business process: select context, choose invoices, review financing, inspect accounting, prepare repayment, and submit. This is a sensible design for banking operations because it prevents the user from jumping directly to submission without seeing the financial consequences of the request.

The service layer follows the same approach used by the rest of the frontend. \texttt{invoiceDisbursementService} is responsible for retrieving invoices that may be financed. \texttt{financeDisbursementService} is responsible for creating the finance disbursement request. Both services are expected to use the shared API client, so authentication, token handling, and gateway routing remain consistent with other modules.

\subsection{Backend Implementation}
\label{subsec:ch5_backend_implementation}

The backend implementation must remain aligned with the SCF service style described in Chapter~\ref{chap:scf_domain}. It should validate the selected program, verify invoice eligibility, compute or confirm financial terms, prepare fee information from the program fee catalogue, create accounting and repayment details, persist the request, and expose lifecycle actions under maker/checker control.

\subsection{Key Technical Decisions}
\label{subsec:ch5_technical_decisions}

Several design decisions are worth recording because they connect the academic module with the real platform:

\begin{itemize}
    \item the module uses a wizard to match the natural order of the business operation;
    \item request creation is separated from approval, preserving maker/checker control;
    \item accounting and repayment information are displayed before submission, not hidden after it;
    \item the API contract follows the existing \texttt{/scf/} gateway convention;
    \item the module reuses program, invoice, fee, and transaction concepts rather than creating an isolated domain vocabulary.
\end{itemize}

\section{Conclusion}
\label{sec:ch5_conclusion}

This chapter presented the Disbursement Module as the academic PFE project. The module sits at a decisive point in the SCF chain: after product, program, invoice, and fee configuration, but before accounting, repayment, transaction follow-up, and approval. The work combined target design, visible frontend flow, service-contract preparation, and Product Owner refinement while the main platform was still progressing toward native integration.
